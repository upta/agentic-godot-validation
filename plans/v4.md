## Plan: V4 Scenario Breadth And Batch Execution

Extend the v3 evidence loop with a small batch runner and a broader scenario set. The aim is to prove that the current generic assertion model, signal probes, and artifact manifests hold up across more than isolated single-step interactions.

The key v4 adjustment is to avoid adding only near-duplicates of the existing canonical scenarios. V4 should still add a stronger movement follow-up and a stronger pause-menu follow-up, but it should also add one clearly different integration scenario that composes gameplay and UI behavior in the same harness.

**Goals**
1. Add a `tools/run_all_scenarios.ps1` wrapper so the retained scenario set can be run in one command.
2. Add a movement scenario that proves motion stops when input is released, not just that motion starts.
3. Add a pause-menu scenario that proves resume activation closes the menu, not just that opening the menu focuses the button.
4. Add one unique composed scenario that exercises gameplay and modal UI together instead of validating them in isolation.
5. Keep the verifier surface on `assert_value` and `assert_pipeline`; prefer better harness state and better scenario design over new bespoke assertion ops.

**Non-Goals**
1. Do not add Playwright, web export, CI, screenshot diffing, or external orchestration.
2. Do not replace the current local runner or artifact contract.
3. Do not add a large general-purpose input scripting system.
4. Do not blur the game/test boundary; production code stays reusable and test-only composition stays under `test/`.

**V4 Contract**
1. `tools/run_all_scenarios.ps1` should enumerate scenario contracts under `test/scenarios/`, invoke `tools/run_scenario.ps1` for each one, and return non-zero if any scenario fails.
2. The batch runner should rely on the existing artifact flow so `artifacts/index.json` and per-scenario `latest.json` remain the canonical retained summary.
3. The new movement scenario should use multiple checkpoints and `assert_pipeline` to prove that displacement stops increasing after release.
4. The new pause-menu completion scenario should validate close behavior through reusable state and signal facts, not a pause-menu-specific verifier rule.
5. The unique v4 scenario should use a combined harness that composes the player actor and pause menu together while preserving the one-way dependency from test code into game code.
6. If keyboard activation of the focused resume button is required, the bootstrap input contract may grow one additional action mapping such as `ui_accept` rather than adding a new driver step op.

**Scenario Package**
1. `move_up_release_stops_motion.json`
   Press `move_up`, wait, release, wait again, and prove that upward displacement increases during the pressed interval but remains stable after release.
2. `pause_menu_resume.json`
   Open the pause menu, activate the focused resume button, and prove the menu closes and related signals reflect a completed interaction rather than only an opened state.
3. `modal_pause_blocks_movement.json`
   Compose the player actor and pause menu in one harness, open the menu, hold movement input while the modal is active, prove actor position does not change while paused, then resume and prove gameplay state is restored. This is the intentionally distinct v4 scenario because it validates cross-system coordination instead of a single isolated component.

**Expected Production And Harness Additions**
1. Extend `game/scripts/ui/simple_pause_menu.gd` with a small completion contract for resume activation and close behavior.
2. Add any minimal production signals needed for reusable validation, such as `resume_requested` or `menu_closed`, if the existing open signal is not enough.
3. Keep composition-specific logic in new test harnesses and controllers rather than expanding `game/scenes/main.tscn` purely for validation.
4. Add a combined player-plus-menu harness under `test/harnesses/` and a matching controller under `test/scripts/harness_controllers/`.

**Implementation Steps**
1. Lock the v4 plan in repo docs.
   Add this plan as the v4 reference before implementation starts.
2. Add the batch runner.
   Wrap `tools/run_scenario.ps1` so the full retained scenario set can be executed from one entrypoint.
3. Extend the pause-menu completion contract.
   Add the smallest production hooks needed so resume can be activated and observed generically.
4. Add the new harness composition.
   Build a combined harness that exposes player, menu, and signal state through the existing `harness_state` conventions.
5. Add the three new scenarios.
   Prefer checkpoint-driven facts and pipeline math over any new verifier-specific behavior.
6. Validate the broadened suite.
   Run the batch runner, inspect retained manifests, and confirm the new integration scenario writes useful failure evidence if broken.

**Verification**
1. Run `tools/run_all_scenarios.ps1` and confirm it executes every scenario under `test/scenarios/`.
2. Confirm the batch runner exits non-zero if any scenario fails.
3. Inspect the new movement summary and verify it records both the moving interval and the stable post-release interval.
4. Inspect the new pause-menu completion summary and verify it records close-related state and signal facts.
5. Inspect the unique integration scenario summary and verify it records unchanged actor position while paused plus restored state after resume.
6. Confirm `artifacts/index.json` and per-scenario `latest.json` point to the latest retained run for all scenarios in the broadened suite.

**Decisions**
- Prefer a third scenario that composes systems together over adding another isolated movement or menu check.
- Prefer minimal production hooks and richer harness composition over expanding the verifier API.
- Treat batch execution as part of the local validation contract before considering any CI surface.
