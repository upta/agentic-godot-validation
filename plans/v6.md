## Plan: V6 Packaging And Reuse

Turn the current proof-of-concept repository into a reusable validation kit that can be dropped into another Godot project with minimal rewriting. The focus shifts from adding validation capability to productizing the existing capability: separating portable runtime code from host-project assets, introducing a host integration layer that does not take over the project's main entry flow, and adding consumer-facing docs plus agent instructions and skills.

V6 should treat the current repository as three things that are currently mixed together:

1. a reusable validation framework
2. a host-project integration layer
3. an example game and sample scenario set

The package boundary should make those distinctions explicit.

**Goals**
1. Separate portable validation runtime code from project-specific harnesses, scenarios, and sample gameplay.
2. Reorganize the repo so the reusable layer can be copied into another Godot project without also copying the current POC game as if it were framework code.
3. Remove the current assumption that the reusable layer owns the host project's main scene or root router.
4. Introduce consumer-facing installation and authoring docs that are smaller and clearer than the internal v1-v5 evolution history.
5. Add workspace-level Copilot instructions and focused skills for installing the validation layer, authoring harnesses and scenarios, and debugging failures.
6. Preserve the current scenario format, artifact model, and tooling contract as much as possible so packaging does not force a verification redesign.

**Non-Goals**
1. Do not add CI, publishing automation, or marketplace packaging yet.
2. Do not redesign the verifier, scenario driver, or artifact schemas unless packaging forces a small path update.
3. Do not treat the current sample gameplay systems as universal framework components.
4. Do not require host projects to adopt the current `bootstrap/app_root.*` pattern as-is.
5. Do not build an in-editor Godot plugin UI in this phase unless a minimal plugin shell becomes necessary for discovery.

**V6 Contract**
1. The reusable validation runtime should live under a dedicated package namespace, preferably `addons/agentic_godot_validation/`.
2. Host-project validation assets should move out of the generic `test/` root and into a clearly project-owned root such as `validation/`.
3. The current POC gameplay and sample validation content should move under `examples/` so consumers can study it without mistaking it for required package structure.
4. The package should provide a documented host adapter pattern instead of owning `project.godot` main-scene routing.
5. Root-level automation that must remain visible to Copilot and developers in a host project should stay at the workspace root, including `tools/` and `.github/` assets.
6. Consumer docs should live under `docs/` and become the primary onboarding surface; `plans/` remains the internal design log.
7. Agent customization should include both always-on guidance and task-scoped skills.

**Target Package Model**
1. Framework-owned
   Portable runtime code, reusable tooling, canonical specs, and package docs.
2. Host-owned
   Harness scenes, harness controllers, project-specific scenarios, and any small integration code that exposes semantic runtime facts from that host project.
3. Example-owned
   The current player movement, pause menu, sample harnesses, and the five existing scenarios as a reference implementation.

**Reorganization Rules**
1. `addons/agentic_godot_validation/` contains only portable runtime code and reusable integration templates.
2. `validation/` is the host-facing workspace where teams author harnesses, host adapters, and scenario contracts for their own game.
3. `examples/minimal_poc/` holds the current sample game and sample validation content as reference material.
4. `docs/` explains how to use the package; `plans/` explains how the package evolved.
5. `.github/` contains the agent-facing usage model that ships with the package overlay.

**Host Integration Model**
1. Replace the current hard-coded root-router assumption with a documented host adapter contract.
2. The host project remains responsible for its own `project.godot`, main scene, and runtime scene graph.
3. The package should ship a small template router or entry scene that a host project can adopt or adapt for `--test-mode`, but the template is not the framework itself.
4. The host adapter is the only place where package code is allowed to bridge between the host's startup flow and the reusable validation runtime.
5. Package docs must explain the minimum required host wiring: test-mode routing, input-map expectations, artifact path injection, and scenario path lookup.

**Agent Assets**
1. `.github/copilot-instructions.md`
   Always-on high-level guidance: preserve the package boundary, keep runtime code generic, keep host assets under `validation/`, and do not move example content back into the framework.
2. `.github/instructions/validation-runtime.instructions.md`
   File-scoped guidance for `addons/agentic_godot_validation/**` focused on portability, API stability, and avoiding host-specific assumptions.
3. `.github/instructions/validation-assets.instructions.md`
   File-scoped guidance for `validation/**` focused on harness design, semantic state exposure, and using `assert_value` and `assert_pipeline` instead of bespoke verifier rules.
4. `.github/skills/install-agentic-godot-validation/SKILL.md`
   Workflow for dropping the package into an existing Godot project, wiring the host adapter, and validating the first scenario run.
5. `.github/skills/author-validation-scenario/SKILL.md`
   Workflow for creating a new harness, controller, and scenario contract in a host project using the package conventions.
6. `.github/skills/debug-validation-failure/SKILL.md`
   Workflow for reading `summary.json`, suite artifacts, screenshots, and signal facts to diagnose failing scenarios.

**Consumer Docs Package**
1. `docs/install-into-existing-project.md`
   What folders to copy, how to wire test mode, how to run the first scenario, and how to confirm artifacts are produced.
2. `docs/host-integration.md`
   How the host adapter works, what remains host-owned, and what the package expects from the project.
3. `docs/write-a-harness.md`
   How to expose semantic node facts, metrics, signal facts, and project-specific state from a new harness.
4. `docs/write-a-scenario.md`
   How to author `assert_value`, `assert_pipeline`, checkpoints, and suite runs.
5. `docs/inspect-artifacts.md`
   How to interpret single-scenario artifacts and suite artifacts, including failure-first summaries and flaky detection.
6. `docs/package-layout.md`
   What belongs in `addons/`, `validation/`, `examples/`, `.github/`, and `tools/`.

**Implementation Steps**
1. Lock the v6 plan in repo docs.
   Add this plan as the packaging and reuse reference.
2. Freeze the target package boundary.
   Confirm what is framework-owned, host-owned, and example-owned before moving files.
3. Move the portable runtime into `addons/agentic_godot_validation/`.
   Preserve current behavior while updating preloads and paths.
4. Create the host-owned `validation/` root.
   Establish the default place for host adapters, harnesses, controllers, and scenarios.
5. Recast the current repo content as an example project.
   Move sample game code and the current five scenarios into `examples/minimal_poc/`.
6. Replace the root-router assumption with a host adapter template.
   Keep test-mode support but make it opt-in and host-owned.
7. Write the consumer docs.
   Produce installation, authoring, artifact-reading, and layout docs.
8. Add the Copilot instructions and skills.
   Ship one always-on instruction layer and task-scoped skills for install, authoring, and debugging.
9. Validate the packaged layout.
   Copy the overlay into a second Godot project or a fresh minimal host project and prove that a sample scenario runs with the documented setup steps only.

**Verification**
1. Confirm the reorg leaves the portable runtime runnable from its new package namespace.
2. Confirm the host-owned `validation/` root can hold project-specific harnesses and scenarios without modifying framework code.
3. Confirm the example POC still runs from `examples/minimal_poc/` after path updates.
4. Confirm the package can be overlaid into a second Godot project using only the consumer docs.
5. Confirm the Copilot instructions and skills can guide install, authoring, and failure triage without requiring the user to read the internal v1-v6 plan history first.

**Decisions**
- Package the runtime as a reusable addon-style namespace, but keep developer-facing tools and agent assets at the workspace root.
- Rename host-facing validation assets from `test/` to `validation/` so they do not get conflated with unit tests or framework internals.
- Treat the current gameplay and scenarios as example content, not as required framework structure.
- Prefer a documented host adapter template over a framework-owned root router.
- Split consumer docs from internal plan history so other teams can adopt the package without reading the entire design evolution.
