## Plan: V3 Evidence Loop Tightening

Extend the v2 generic model in four specific ways: named-source pipeline assertions, reusable signal probes, artifact manifests, and failure-first summaries. The goal is to make the POC more reusable without changing the game/test boundary or introducing any new external tooling.

**Goals**
1. Add `assert_pipeline` so scenarios can compare derived values without repeating contract literals.
2. Expose reusable signal probe facts under `harness_state.signals` for interaction checks that are better modeled by signal emission than by static state.
3. Rebuild `artifacts/index.json` and `artifacts/<scenario_id>/latest.json` after pruning so the latest retained evidence is easy to discover.
4. Surface a top-level `failed_assertion` block in run summaries so the first failing check is immediately visible.

**Non-Goals**
1. Do not change the game/test boundary.
2. Do not add Playwright, web export, CI, screenshot diffing, or cloud orchestration.
3. Do not replace `assert_value`; keep it as the simple direct-check primitive.
4. Do not add a large library of scenario-specific verifier operations.

**V3 Contract**
1. `assert_pipeline` resolves named sources from checkpoints, the scenario contract, or literals.
2. `assert_pipeline` supports a small initial transform set: `add`, `subtract`, and `abs`.
3. `assert_pipeline.assert.actual` references a named source or computed output.
4. `assert_pipeline.assert` compares against either `expected_source` or a literal `expected` value.
5. Harness signal facts live under `harness_state.signals.<alias>.<signal_alias>`.
6. Signal facts are count-oriented and currently expose `count`, `connected`, `signal_name`, `source_path`, and `last_emitted_msec`.
7. `summary.json` surfaces `failed_assertion` at the top level and within the final `result` object when a run fails on an assertion-related check.
8. `tools/prune_artifacts.ps1` is responsible for rebuilding artifact manifests after retention is applied.

**Canonical Scenarios**
1. `move_up_smoke.json` should use `assert_pipeline` with checkpoint Y-values and a contract-sourced threshold.
2. `pause_menu_focus.json` should continue to use `assert_value` for semantic node facts and add signal count assertions for the reusable probe layer.

**Implementation Steps**
1. Lock the v3 schema in repo docs.
   Update the scenario and artifact specs and add this plan as the v3 reference.
2. Add the signal probe helper and wire the pause-menu harness into it.
   Emit a production `menu_opened` signal, capture `menu_opened` and `focus_entered`, and expose the resulting facts through harness state.
3. Extend the driver and verifier for `assert_pipeline`.
   Keep `assert_value` intact, add named source resolution, add a small transform set, and preserve the existing result flow into summaries.
4. Enrich assertion failures.
   Attach step metadata to verifications and surface a top-level `failed_assertion` block with related checkpoints and screenshots.
5. Add artifact manifests.
   Rebuild `artifacts/index.json` and `artifacts/<scenario_id>/latest.json` after pruning so retained evidence stays discoverable.
6. Prove the updated loop.
   Run the canonical scenarios, inspect the generated manifests, and confirm that an intentional failure writes a useful `failed_assertion` block.

**Verification**
1. Run `move_up_smoke.json` and confirm pass.
2. Run `pause_menu_focus.json` and confirm pass.
3. Inspect `artifacts/index.json` and the per-scenario `latest.json` files.
4. Run an intentional failing assertion and confirm `summary.json` surfaces `failed_assertion` with related screenshots.

**Decisions**
- Keep `assert_value` for direct checkpoint reads and use `assert_pipeline` when contract-sourced math improves reuse.
- Keep the first signal probe layer intentionally simple and count-based.
- Generate manifests from pruning instead of adding a second artifact-indexing step to the runner.
