## Plan: V1 Godot Evidence Loop

Build a desktop-only, local-only, single-scenario vertical slice in Godot 4.6 with a strict separation between gameplay code and validation code. Keep production scenes and scripts under a dedicated game root, keep harnesses, drivers, inspectors, and scenarios under a dedicated test root, and allow only one-way dependencies from test into game. Use a Godot-native GDScript test layer to launch a harness scene, inject deterministic input, collect runtime facts and screenshots, and write a verifier-friendly artifact bundle. Defer web/Playwright, CI, generalized agent integration, and screenshot diffing until the loop works with one trivial movement case.

Implementation note for steps 1 through 4: the locked contract now lives in `/plans/specs/v1_contracts.md` and the machine-readable scenario contract lives in `/test/scenarios/move_up_smoke.json`. Because Godot exposes only one `run/main_scene`, the project uses a neutral router under `/bootstrap/app_root.tscn` that selects either `/game/scenes/main.tscn` or `/test/bootstrap/test_bootstrap.tscn` based on `--test-mode` while keeping `game/` free of any dependency on `test/`.

**Steps**
1. Lock the v1 contracts.
   Define the done contract for the first scenario as: pressing W moves the controlled sprite upward by at least a fixed delta within a fixed frame budget.
   Define the artifact contract: summary.json, event_log.json, scene_tree.json, console.log, screenshots/before.png, and screenshots/after.png are all required. Missing any file is an automatic failure.
   Define the exit code contract: 0 pass, 1 assertion failure, 2 runtime error, 3 timeout, 4 artifact generation error.
2. Establish the project boundary rules. Depends on step 1.
   Split the project into a production game root and a test root, for example game/ for shipped scenes and scripts and test/ for harnesses, drivers, inspectors, and scenarios.
   Enforce a one-way dependency rule: test code may load and observe game code, but game code must never import, call, or reference test code.
   Keep all determinism toggles and test-mode setup in a single test bootstrap layer rather than scattering test conditionals through gameplay scripts.
3. Create the minimal test entrypoints. Depends on step 2.
   Add one dedicated test bootstrap scene or a single test bootloader autoload enabled by CLI args. The current implementation uses a dedicated test bootstrap scene selected by a neutral root router because `project.godot` can only point to one main scene.
   Standardize a CLI convention such as: godot --path <project> -- --test-mode --scenario <path> --artifacts <dir>.
   Freeze deterministic settings for test runs at bootstrap time: fixed RNG seed, fixed physics tick, fixed window size, muted audio, and no nonessential background systems.
4. Build the first deterministic harness. Depends on step 3.
   Create a tiny harness scene under the test root with one controllable actor or sprite, no camera complexity, no unrelated UI, and a fixed starting transform.
   Reuse production movement code by instantiating the real gameplay actor or controller from the game root rather than duplicating logic in the harness.
   Add a harness controller script that can reset state on every run and expose current position to the runtime inspector.
5. Implement the in-engine scenario driver. Depends on step 3. Parallel with step 6.
   Use a minimal JSON scenario format with only the primitives needed for v1: load_harness, checkpoint, press_action, release_action, wait_frames, assert_position_delta, and quit.
   Drive the game from inside Godot using synthetic input such as Input.action_press or Input.parse_input_event rather than Playwright.
   Record the executed step log with frame numbers in event_log.json.
6. Implement the runtime inspector v1. Depends on step 3. Parallel with step 5.
   Expose runtime facts through observation only: node existence, current scene, visibility, focused control, selected node paths, world position for the test actor, error and warning capture, tree snapshots, and screenshot capture.
   Do not add test-only dependencies into gameplay classes; prefer public methods, standard node APIs, and existing signals wherever possible.
   Write summary.json and scene_tree.json at checkpoints and again at scenario end.
7. Implement the local runner wrapper. Depends on steps 4 and 5.
   Add a Windows-first PowerShell wrapper that creates a run directory, launches Godot with the scenario args, waits for process exit, and validates that all required artifacts exist.
   Write artifacts under artifacts/<scenario>/<timestamp>/ and emit a single machine-readable result line plus the artifact path.
8. Add the first verifier rules. Depends on step 6.
   Evaluate observed Y displacement, missing artifacts, runtime errors, and screenshot presence.
   Fail if the movement threshold is not met even if the input step executed successfully.
   Keep verifier rules data-driven so the planner and verifier agents can read the exact contract later.
9. Prove the loop end to end. Depends on step 7.
   Run the same movement scenario at least 10 consecutive times on the same machine and confirm stable results.
   Introduce one intentional failure, such as disabling movement or changing the input mapping, and confirm that the system fails with useful evidence.
   Do not widen scope until this loop is stable.
10. Expand to the second interaction. Depends on step 8.
   Reuse the same driver, inspector, and artifact contracts for a pause-menu or simple modal-focus scenario.
   Revisit browser export or Playwright only after the native desktop path is proven.

Implementation note for step 10: the second interaction is currently a simple modal-focus scenario triggered by `Escape`, implemented through `/game/scenes/ui/simple_pause_menu.tscn`, `/test/harnesses/pause_menu_harness.tscn`, and `/test/scenarios/pause_menu_focus.json`.

**Boundary Rules**
- Keep shipped gameplay assets under a dedicated game root such as /game/scenes, /game/scripts, and /game/resources.
- Keep validation assets under a dedicated test root such as /test/harnesses, /test/scripts, and /test/scenarios.
- Allow neutral entry infrastructure under /bootstrap only when it stays outside both roots and does not introduce reverse dependencies from game into test.
- Allow only one-way dependencies from test into game. Game code must be removable from the test layer without compilation or runtime references back into test.
- Activate validation through one explicit test bootloader or bootstrap scene. Do not register broad test autoloads in the normal production path.
- Keep observability read-only: the test layer may query public methods, node state, transforms, and signals, but it should not require special test-only branches in gameplay classes.
- Keep all artifacts under /artifacts and exclude them from normal project content and exports.

**Verification**
1. Run the local runner against the movement scenario and confirm a fresh artifact bundle is created.
2. Run the same scenario 10 times consecutively and confirm the observed displacement stays inside the same threshold band each time.
3. Intentionally break movement and confirm the exit code, summary status, and evidence artifacts all indicate failure.
4. Confirm the runtime inspector reports the expected node path, position values, current scene, and frame numbers at each checkpoint.
5. Rerun after a failure and confirm the next run starts from a clean initial state.

**Decisions**
- Target Godot 4.6 for v1.
- Optimize for local developer machines first. CI is explicitly deferred.
- Use a GDScript-based validation layer even if later gameplay code is written in C#.
- Use a native desktop runner for v1 and do not depend on Playwright for the first proof.
- Preserve a hard separation between game code and validation code: game never depends on test, and test activation happens through one explicit bootstrap path.
- The first scenario is intentionally trivial: prove that a running game session actually moved in response to input and produced evidence.
- Excluded from v1: Playwright or web export, GitHub Actions, screenshot diffing, dialog harnesses, generalized multi-agent orchestration, and multi-platform support.

**Plan Persistence**
- Persist the plan inside the repository, not only in chat memory, so future agent sessions can read and revise the same source of truth.
- Start with a single canonical repository document for the plan so future sessions have one obvious source of truth.
- Treat the current plan file as the initial overview document; add focused spec docs only when sections become normative and need to be referenced independently.
- The first spec docs split out during implementation are boundary rules, the locked v1 contract, the executable scenario format, and the artifact schema. Rollout or roadmap details can still be separated later if they become independently normative.
- Keep vision and rationale in the overview; move machine-relevant contracts into smaller docs only when the implementation starts to solidify.
- If repo-wide agent behavior needs guidance before the system exists, add lightweight workspace instructions that point agents to the persisted plan and boundary rules rather than creating a full skill immediately.
- Create a dedicated skill only after the workflow stabilizes enough that its steps, file locations, and artifact contracts are unlikely to churn.

**Further Considerations**
1. If browser-driven validation is still needed later, add it as a second runner backend while keeping the same scenario and artifact schemas.
2. If the game becomes C#-heavy, keep the validation boundary language-agnostic: gameplay can be C#, while the runner and inspector stay in GDScript.
3. If speed matters more than completeness right now, skip GUT in the first vertical slice and add unit tests only after the scenario evidence loop works once.
